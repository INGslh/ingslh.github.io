4.1 C++还是Python？

在我们开始编写OOT模型之前，这是一个我们每一个人都会考虑的问题。教程3已经解决选择编程语言在OOT模型中。

或者

除了编译器和解释器外，有很多不同之处。要决定编程语言的语言，重要的是我们要限制GNU Radio的不同点。首先，这更多的取决于OOT模块。就性能而言，使用C++会更好一些，如果OOT模块的性能不是主要问题，Python是一个好的选择，因为他简洁明了。而且python还更快一些，因为我们不需要编译测试模块。

注意函数声明还要添加到类标题中。(my_qpsk_demod_cb_impl.h)

函数get_minimum_distances是QPSK的解调器函数，从理论上讲，该函数应该计算每个理想QPSK符号到接收符号的距离(数学上等价于确定接收样本的Voronoi区域)。

对于QPSK信号，这些Voronoi区域仅仅是复杂平面中的四个象限。因此，为了将样本解码成比特，将接收到的样本映射到这些象限是有意义的。

现在让我们观察下forecast()。函数系统需要知道需要多少数据才能确保每个输入数组的有效性。如前所述，forecast()方法提供了这些信息，因此您必须在编写gr::block时重写它。

默认实现的forecast()表示noutput_items与每个输入流的需求之间有1:1的关系。项目的大小由gr .:io_signature定义::在gr::block的构造函数中进行。输入和输出项的大小当然不同;这仍然符合1:1的关系。当然，如果你有这种关系，你不会想用gr::block!

尽管通过函数my_qpsk_demod_cb实现了1:1输入输出关系，但这不适用于插补器，抽取器，还有和noutput_items之间有复杂关系的输入设备。
也就是说，通过从类gr::sync_block，gr::sync_interpolator或者gr::sync_decimator中推导,重新填充私有构造函数并覆盖general_work()和forecast()将满足我们的代码块的编码结构。但在类gr::block中有更具体的功能。这些功能是在高级主题部分中介绍的。

4.3 进阶部分

之前的学习已经使我们基本掌握了独立设计OOT模块。但gnuradio的也不是这么简单。因此这一节上，我们将从QPSK解调器转移到其他很少使用或者拥有更具体功能的模块上。
为了给学习添加更多的实际意义，建议热情的读者通过学习现有模块的源代码，能够更新优化现有的代码，并围绕其进行一系列的开发。
